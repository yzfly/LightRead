# 藏书导入导出协议

LightRead 现在直接使用符合 Google Open Knowledge Format（OKF）v0.1 的开放 bundle，不再以私有 `manifest.json` 作为权威数据源。

导出文件使用 `.okf.zip` 扩展名。解压后得到普通的 Markdown、YAML frontmatter 和原始书籍文件：

```text
library.okf.zip
├── index.md
├── manifest.json                  # 可选兼容清单
├── books/
│   ├── index.md
│   └── <book-id>.md
├── catalogs/
│   ├── index.md
│   └── <catalog-id>.md
└── assets/books/<book-id>/
    ├── content.epub
    └── cover.jpg
```

每本书都是标准 OKF concept：通用字段直接使用 `type`、`title`、`description`、`resource`、`tags` 和 `timestamp`；作者、语言、文件、阅读状态及标注使用厂商无关的 Open Library OKF Profile。即使其他软件不实现完整 Profile，也能通过标准 OKF 字段和 Markdown 正文读取书目信息。

同级的 `manifest.json` 是可选的非权威兼容清单，用于旧版 LightRead 工具快速读取同一份数据。导入时始终优先解析 OKF `index.md`；两者冲突时以 OKF concept 为准。第三方软件可以完全忽略它，第三方生成的纯 OKF 包也不需要提供它。

权威的英文 Profile 见 [library-okf-profile.md](./library-okf-profile.md)。它完整定义了字段、资源寻址、摘要校验与导入语义，其他软件不依赖 LightRead 代码或账号即可实现。

## 兼容性

- 当前导出：OKF v0.1 + LightRead Library Profile 1.0，并附带可忽略的 JSON v2 兼容清单。
- 当前导入：支持本 Profile，也接受其他生产者生成的 `Book`、`Publication`、`Paper` 或 `Article` concept，只要 concept 指向 bundle 内受支持的书籍文件。
- 历史导入：继续支持仅含 LightRead JSON manifest v1 或 v2 的旧备份。

## 为什么这样使用 OKF

OKF 本身只规定 Markdown、YAML frontmatter、concept、索引和链接等互操作表面，不规定藏书领域字段。Open Library Profile 是 OKF 允许的公开扩展，而不是私有二进制协议：

- 一个普通 Markdown/YAML 工具即可读取；
- 文件资源使用相对路径，不经过 LightRead 服务；
- SHA-256、字节数和 MIME 类型均公开；
- 标注同时存在于结构化 YAML 与可读 Markdown 正文中；
- 书源账号和密码绝不导出。
