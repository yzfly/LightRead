# LightRead 藏书交换格式

状态：v2，随 LightRead 1.1.x 引入。

## 目标

LightRead Library Archive（扩展名 `.lightread`）用于在 LightRead 实例之间完整迁移藏书。它需要同时承载：

- 原始书籍与论文文件；
- 书名、作者、语言、标签和来源等元数据；
- 阅读位置、进度、累计阅读时间与置顶状态；
- 书签、划线和笔记；
- 不含登录凭据的自定义书源。

它不是新的电子书正文格式，也不尝试替代 EPUB、PDF 或其他书籍格式。

## 容器结构

`.lightread` 是采用 ZIP 物理封装的单文件容器。书籍载荷通常已经压缩，因此 ZIP 条目使用 store 模式，避免重复压缩。

```text
library.lightread
├── manifest.json
└── books/
    └── <archive-book-id>/
        ├── content.<原始格式>
        └── cover.jpg                 # 可选
```

所有路径均相对于容器根目录。`manifest.json` 必须使用 UTF-8 编码。

## manifest.json

顶层结构：

```json
{
  "format": "org.lightread.library",
  "version": 2,
  "exportedAt": "2026-07-22T08:00:00.000Z",
  "generator": {
    "name": "LightRead",
    "version": "1.1.0"
  },
  "books": [],
  "annotations": [],
  "sources": []
}
```

每个 `books` 元素包含 LightRead 的 `BookMeta` 字段，以及一项必需的 `content` 资源和一项可选的 `cover` 资源：

```json
{
  "id": "4f7f…",
  "title": "示例书",
  "author": "示例作者",
  "format": "epub",
  "fileName": "example.epub",
  "language": "zh-CN",
  "tags": ["小说", "待读"],
  "addedAt": 1784707200000,
  "progress": 0.42,
  "location": "epubcfi(…)",
  "hasCover": true,
  "kind": "book",
  "content": {
    "path": "books/4f7f…/content.epub",
    "mediaType": "application/epub+zip",
    "byteLength": 123456,
    "sha256": "64 位小写十六进制摘要"
  },
  "cover": {
    "path": "books/4f7f…/cover.jpg",
    "mediaType": "image/jpeg",
    "byteLength": 12345,
    "sha256": "64 位小写十六进制摘要"
  }
}
```

资源描述符的四个字段在 v2 中均为必需字段。`sha256` 对 ZIP 解包后的条目字节计算，而不是对压缩数据计算。

`annotations` 与书籍通过导出包内的 `bookId` 关联。导入后实现会建立新 ID 映射，不要求目标库沿用包内 ID。

`sources` 只保存公开的书源描述；`username` 和 `password` 不得写入 v2 包。容器本身不加密，因此使用者仍应像对待原始电子书一样保管它。

## 导入语义

1. 识别 `format` 和 `version`，未知版本直接拒绝，避免误读。
2. 在写入数据库前检查每个必需载荷的存在性、字节数和 SHA-256；任一失败则不开始导入。
3. 以 `(title, fileName)` 识别目标库中已有书籍。已有书籍不覆盖，缺失的标注会合并进去。
4. 标注使用类型、位置、文本、笔记、颜色和创建时间组成的内容指纹去重，因此重复导入是幂等的。
5. 自定义书源按 URL 去重；内置书源由应用版本管理，不从包中覆盖。
6. 导入器继续接受旧的 `app: "lightread", version: 1` ZIP 备份，但旧格式没有完整性摘要。

## 为什么不直接使用 Google OKF

Google Cloud 在 2026-06-12 发布的 OKF v0.1 是一个草案，目标是把供人和 AI 代理读取的“知识”表示为带 YAML frontmatter 的 Markdown 目录。它只强制 concept 的 `type` 字段，并允许生产者自由扩展；官方规范也把“替代领域专用 schema”列为非目标。

这使 OKF 很适合为藏书生成可浏览、可检索、可交给 AI 的说明层，但不适合作为 LightRead 的规范导入格式：

- 它没有定义二进制书籍载荷的打包与寻址；
- 它没有内容长度、摘要或损坏检测约定；
- 它没有阅读进度、标注、冲突合并等藏书领域语义；
- v0.1 仍是 draft，直接依赖会把应用迁移兼容性绑在快速变化的规范上。

因此 v2 采用“ZIP + 版本化 JSON manifest + 原始载荷”。未来若需要 AI/知识库互操作，可以在容器中增加可选的 `okf/` sidecar：一书一份 Markdown concept，并把 `resource` 指向对应载荷；`manifest.json` 仍是导入时的唯一权威数据源。
